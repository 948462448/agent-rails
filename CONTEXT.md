# Agent Rails Context

Agent Rails is a personal, repository-independent kit. It reads a target project through `--project`; it does not make the target project part of this repository.

## Core vocabulary

- **Target Project**: The repository or worktree Agent Rails inspects and equips. Its root and profile must be resolved for every invocation.
- **Target Project Context**: The canonical Target Project root, Git presence, resolved Profile, effective project name, worktree slug, and derived local Task Pack path for one invocation.
- **Local Adapter**: Tool-specific files that connect Claude Code or OpenCode to Agent Rails inside a Target Project. Local Adapters are personal artifacts and must not become team defaults accidentally.
- **Managed Artifact**: A file or skill directory owned by Agent Rails and therefore safe for Agent Rails to refresh or remove.
- **Generated File**: A Managed Artifact identified by the current generated marker or a supported legacy signature.
- **Generated Adapter Content**: The rendered guide, Claude project block, and pack/lite/check command text written as Managed Artifacts for one Local Adapter.
- **Managed Skill Inventory**: The adapter-local list of skill directory names Agent Rails installed. It is the source of truth for safe refresh and uninstall behavior.
- **Managed Adapter Workspace**: The shared Module that applies ownership, tracked-path protection, managed-skill, and local-ignore rules while installing, refreshing, or removing a Local Adapter.
- **Profile**: Configuration resolved for one Target Project or worktree. A Profile must not leak into sibling repositories.
- **Task Pack**: A generated, task-scoped context artifact consumed by an agent run.
- **Pack Mode**: The evidence-density policy for a Task Pack. Lite, Normal, and Deep bound repeated excerpts while preserving capability sections; Audit keeps the Profile's configured maxima.
- **Model Preset**: A canonical model identity with aliases, context limits, throughput limits, and Pack Mode token budgets shared by all consuming commands.
- **Verification Plan**: The ordered, de-duplicated verification commands selected from a changed-path snapshot. Check owns reporting and execution policy; Task Pack and publish integrations consume the same read-only Plan Interface.
- **Verification Failure**: The failed Plan step, exit status, completed-step count, and bounded complete-line stdout/stderr evidence returned by Agent Check without changing the original live stream or exit semantics.
- **Repair Pack**: The bounded, redacted failure summary emitted by Verify after a non-zero Verification Failure. It identifies the first high-value diagnostic, confirms locations present in the frozen changed-path scope, and may retrieve related tracked code locations from the same fixed Git target; it does not claim a root cause.
- **Agent Check Application Service**: The Python orchestration boundary that resolves one Target Project/Profile, freezes one Git Scope and Verification Plan, renders the public report, and optionally executes opaque verification commands through an isolated child shell.
- **Publish Check Application Service**: The Python read-only orchestration boundary that resolves one Target Project/Profile, freezes Git and repository metadata, composes one Verification Plan, scans the four publish-change layers, and renders deployment-baseline evidence.
- **Doctor Application Service**: The Python diagnostic boundary that resolves one Target Project/Profile, reports kit/tool/adapter/memory/Git health, and optionally composes the Claude Adapter Application Service for explicit repair.
- **OpenCode Adapter Application Service**: The Python orchestration boundary that resolves one Target Project/Profile and composes Adapter Content, Managed Adapter Workspace, plugin-template rendering, user-config mutation, and install/doctor/uninstall output policy.
- **Claude Adapter Application Service**: The Python orchestration boundary that resolves one Target Project/Profile and composes Adapter Content, Managed Adapter Workspace, marked project rules, optional personal reminders, SessionStart settings, and install/uninstall output policy.
- **Codex Adapter Application Service**: The Python orchestration boundary that registers, diagnoses, or removes the kit-local Codex plugin, resolves optional Target Project identity without executing its Profile, and composes explicit project repair through Doctor.
- **Run Facade**: The Python User Journey boundary that resolves one Target Project/Profile, prepares one Task Pack through the Pack Application Service, and estimates that exact artifact through the shared Tokenizer Interface.
- **Setup Facade**: The Python User Journey boundary that resolves one Target Project/Profile and composes deliberate Adapter install and Doctor requests around the same Context without recursively invoking the public CLI.
- **Verify Facade**: The Python User Journey boundary that resolves one Target Project/Profile, executes Agent Check, and only after success optionally composes a fresh Publish Check snapshot around the same Context.
- **Git Scope**: The resolved target commit, optional base commit, merge base, and committed/worktree path snapshot used by Task Pack, Agent Check, and publish checks.
- **Task Pack Change Evidence**: The ranked changed-path view, diff-first redacted excerpts, Git-state Markdown sections, and raw changed-path inventory composed from Git Scope for downstream Task Pack consumers.
- **Code Evidence Module**: The shared, bounded read-only retrieval Module that selects tracked code, test, symbol, and line locations from one immutable Git target using literal query terms. Task Code Evidence and Repair Pack are separate Adapters at this Interface.
- **Task Code Evidence**: The bounded, goal-aware tracked-file, symbol, and test-location evidence selected from a clean Git target snapshot when changed-path evidence is unavailable.
- **Task Pack Project Docs**: The changed-path-aware Entry Docs selection, Context Gaps, and Project Configuration status composed for one Task Pack, including explicit-target lookups through isolated Git context.
- **Task Pack Memory Evidence**: The budgeted local-card and optional Online Memory evidence composed from the goal and NUL-safe changed-path snapshot for one Task Pack.
- **Task Pack Contract Sections**: The stable Agent Rails, subagent-result, and delivery contracts rendered from Profile rules for one Task Pack.
- **Final Task Pack Renderer**: The Python Module that composes the fixed Task Pack section order, applies the optional hard token cap, and atomically publishes one private artifact.
- **Task Pack Markdown Interface**: Shared display-text, code-span, fence, and valid-UTF-8 primitives used by context Modules so untrusted values cannot change Pack structure.
- **Task Pack Application Service**: The Python orchestration boundary that resolves one Target Project/Profile, applies CLI overrides, invokes each context Module once in memory, freezes explicit target commits, and publishes the final Pack.
- **Memory Suggestion Application Service**: The Python orchestration boundary that records one local memory decision and optionally publishes one provider-neutral local card without invoking Online Memory or loading its credentials.
- **Private Text Publisher**: The shared Module that stages complete UTF-8 private artifacts, enforces mode `0600`, rejects unsafe targets, and publishes each file atomically while reporting any committed prefix on a later failure.
- **Sensitive Output Guard**: The shared Module that detects supported secret-bearing assignments, headers, and private-key blocks, then applies the evidence policy required by each Adapter.
- **Terminal Output Module**: The shared Module that renders untrusted literal/stream text and line/chunk events without allowing terminal control injection. The standalone Release Installer remains self-contained at its cold-start seam.
- **Adapter Output Module**: The shared Module that gives Claude, Codex, and OpenCode one terminal-safe stdout/stderr event Interface, error history, and result rendering while their actions and lifecycle policy remain tool-specific.
- **Child Process Module**: The shared Module that concurrently decodes configured child streams, closes their pipes, and boundedly terminates the process groups used by Check and Update.
- **Release Bundle**: The versioned, Git-metadata-free kit archive plus SHA-256 digest and standalone installer published as one GitHub Release.
- **Release Install**: A versioned local kit directory selected through an atomic `current` symlink, with a stable user-level CLI entrypoint.
- **User Journey Facade**: A small public command that composes existing Interfaces around one user goal without taking ownership of their domain rules. Setup, Run, and Verify form the default journey.
- **Test Suite**: A domain-grouped set of regression tests loaded by the test runner. Suites share assertions and temporary-workspace setup but own their test cases and execution labels.
- **Related Test Selection**: The changed-path mapping from one refactor slice to the smallest safe ordered set of Test Suites. It accelerates local feedback without replacing the full pre-delivery regression gate.
- **Context Budget Assembler**: The Python component that selects and truncates Task Pack sections under a hard token cap using one Tokenizer Interface.
- **Trusted Python Bootstrap**: The absolute-path launcher invoked with Python environment isolation that inserts this kit's `src` directory ahead of the caller's working directory before importing the Python CLI, preventing a Target Project from shadowing `agent_rails` or injecting `sitecustomize` while preserving relative-path behavior.
- **Public CLI Dispatcher**: The Python command tree behind the top-level bootstrap. It owns public help/version/home behavior, nested-command validation, exact isolated Python-helper argv/environment, and the legacy `--project` working-directory seam without routing through Shell.
- **Update Application Service**: The Python orchestration boundary that selects a Git checkout or Release source, applies clean/fast-forward and skip policy, and composes source tests plus exact tool-specific Doctor and Adapter child commands. Release replacement is a process boundary, so accumulated output must be flushed before re-exec.
- **Online Memory Adapter**: A provider-neutral, read-only search Adapter invoked as an external command. It owns its credentials and provider protocol, reads query metadata from the documented environment, and returns UTF-8 Markdown on stdout.
- **External Evaluation Harness**: Standalone tools that capture completed TUI artifacts, perform blind comparison, and normalize optional trajectories without becoming an Agent Rails runtime command.
- **Host Bootstrap Shell**: One of the four irreducible operating-system seams: the symlink-aware CLI entrypoint, two SessionStart host wrappers, or the standalone Release cold-start installer. It owns no product policy.

## Architectural boundaries

- Adapter entrypoints own tool-specific CLI behavior, paths, configuration merges, and reminder or instruction blocks.
- The shared Managed Adapter Workspace Module owns Generated File recognition, Managed Skill Inventory mechanics, tracked-path protection, generated-file writes, managed-skill installation/removal, local Git exclude resolution, and local-ignore block lifecycle. Project writes and recursive skill copies/removals are anchored through no-follow directory handles; its Interface preserves tool-specific entries and output policy without duplicating workspace mutation rules in each Adapter.
- The Python Adapter Content Module owns Generated Adapter Content rendering, tool-specific frontmatter, shell-safe executable/Profile arguments, and lossless Profile metadata for SessionStart. Adapter entrypoints only select the tool, request an artifact, and write it.
- The Task Pack generator owns Pack Mode density caps and must retain the goal, Git state, prioritized changes, Task Code Evidence, entry docs, memory, contract, grill, verification, delegation, and delivery seams in every mode; truncation must preserve complete lines and valid UTF-8. Output is staged beside its destination and atomically replaced only after rendering succeeds.
- Changed File Excerpts are diff-first for tracked paths and prefix-based only for untracked text, so the evidence seam favors changed behavior over file headers.
- Smart sorting may use a bounded set of meaningful goal tokens against paths and actual changes; project-name and workflow-generic tokens do not earn priority.
- The Sensitive Output Guard owns one detection Implementation for Task Pack and publish Adapters. Its redaction Interface stays conservative and fails closed; its scan Interface suppresses recognizable code expressions and can map added diff lines back to source paths and line numbers. Publish scanning composes committed, staged, and unstaged diffs with full untracked-file scans, retaining literal secret and private-key evidence without promoting unchanged tracked content.
- The Python Verification Plan Module owns changed-path matchers, stable command ordering, first-reason de-duplication, target-tree shell-file checks, and suggestion rendering. Agent Check owns full-report and command-execution policy; Task Pack consumes the same Plan directly instead of recursively starting Agent Check.
- The Agent Check Application Service loads its allowlisted Profile once, holds Git Scope and Verification Plan in memory, rejects target or changed-path drift before execution, scrubs inherited repo-local `GIT_*` variables, and invokes each opaque Profile command only as an explicit child-shell argv. It preserves complete live child output while retaining only a bounded tail of complete logical lines for a non-zero Verification Failure; oversized lines are omitted rather than partially retained. The Public CLI enters it through the trusted Python helper.
- The Publish Check Application Service loads its allowlisted Profile once without its environment file, freezes commit SHAs before diff scanning, builds Verification suggestions in process, and scans committed, staged, unstaged, and untracked content through the Sensitive Output Guard. Untracked reads are project-root anchored and no-follow, remote credentials and terminal controls are inert before rendering, and `--no-secret-scan` performs no content reads.
- The Doctor Application Service executes its compatibility Profile and optional environment file in one isolated load, distinguishes their failures, keeps provider diagnostics and online Memory cards out of output, and reads project/kit diagnostic files through bounded no-follow descriptors. Control and direction characters cannot forge diagnostic lines; `--fix` delegates mutation to the Claude Adapter Application Service.
- The OpenCode Adapter Application Service loads its allowlisted Profile once, owns tool-specific layout and config/plugin policy, and composes the shared Adapter Content and Managed Adapter Workspace Interfaces. A preflight Plan validates config, ownership state, strict fingerprinted skill inventory, managed paths, and local-ignore paths before mutation; apply-time atomic rechecks and survivor ownership preserve modified skills and unrelated user config/schema/plugin entries across moves and repeated installs. Local/project and dry-run/force behavior remain public contracts.
- The Claude Adapter Application Service loads its two allowlisted personal-path Profile fields once for install, owns local/project rules and personal settings policy, and composes the shared Adapter Content and Managed Adapter Workspace Interfaces. It validates strict skill ownership, marker balance, JSON shape, SessionStart hook availability, managed paths, and local-ignore writeability before mutation; uninstall removes only proven artifacts and exact marked blocks.
- The Codex Adapter Application Service owns plugin marketplace/add/remove command order, strict action-specific CLI policy, external exit-code propagation, and optional Doctor repair composition. Project inspection never executes the Target Project Profile, marker reads stay below the canonical project through bounded no-follow descriptors, and child diagnostics cannot forge terminal output.
- The shared Git Scope Module owns default-base policy, commit-ref validation, isolated project HEAD resolution, merge-base resolution, and committed/worktree path snapshots. Task Pack, Agent Check, and publish checks are Adapters at this Seam; explicit-target Pack consumers reuse the resolved commit rather than reopening a movable ref.
- The shared Code Evidence Module owns literal query-token selection, fixed-target tracked-path search, lightweight symbol recognition, preferred-path ranking, binary filtering, per-blob size limits, and bounded location records. It never reads worktree-only or untracked source and never returns source bodies. Task Pack Change Evidence and Repair Pack supply their own query and presentation policy as Adapters.
- The Task Pack Change Evidence Module composes Git Scope, the Sensitive Output Guard, and Code Evidence. It owns goal-aware changed-path ranking and the six Git evidence sections, reads only no-follow regular untracked files for changed excerpts, and preserves the raw changed-path inventory for Entry Docs and Memory selection. When the snapshot is clean, its Task Code Evidence Adapter queries the fixed target commit and emits locations and reasons rather than source bodies.
- The Task Pack Project Docs Module consumes the raw changed-path inventory, owns prefix-specific Entry Docs selection and the Entry Docs, Context Gaps, and Project Configuration sections, and prevents configured paths from forging Task Pack sections. Explicit-target existence checks use the shared isolated Git runner.
- The Task Pack Memory Evidence Module owns deterministic direct-card discovery, trigger matching, budget splitting, Sensitive Output redaction, safe Markdown rendering, and non-fatal Online Memory fallback. It composes the provider-neutral Online Memory Interface without taking ownership of provider credentials, protocol, schemas, or writes.
- The Task Pack Contract Sections Module owns stable heading and rule order, Lite Grill guidance, empty-rule policy, and control-character-safe Markdown for the Agent Rails, subagent-result, and delivery contracts. The Pack renderer decides where those independently placeable sections appear.
- The Repair Pack Module owns Sensitive Output redaction, terminal-safe diagnostic selection, changed-scope location confirmation, failure-query construction, and the final character cap for one Verification Failure. It uses the shared Code Evidence Interface to retrieve a few related tracked source/test locations from the frozen verification target, treats retrieval failure as non-fatal, receives facts rather than raw child processes, and never infers a root cause or repeats the opaque Profile command.
- The Final Task Pack Renderer owns the preamble, Session Marker, Goal and Context Budget, the fixed 18-section order, dynamic Verification fences, strict UTF-8, hard-cap integration, and same-directory `0600` atomic replacement. Goal and path values are inert Markdown data, structural validation ignores headings inside fences, and an insufficient hard token budget fails without replacing an existing complete Pack.
- The Task Pack Markdown Interface owns shared visible control-character rendering and collision-free Markdown spans/fences. Evidence, docs, memory, contracts, and the Final Renderer must consume this Interface instead of importing formatting helpers from one another.
- The Task Pack Application Service owns Pack CLI/Profile precedence and stage ordering. It loads the executable compatibility Profile and environment file once through an allowlisted Adapter, resolves Pack Policy once, calls context Modules in process, passes the Evidence Module's full target commit to every explicit-target consumer, treats Verification suggestions as optional, and anchors relative runtime paths to the canonical Target Project.
- The Memory Suggestion Application Service owns its CLI/Profile precedence, canonical memory-card identifiers, Git worktree snapshot, decision/card rendering, Sensitive Output rejection, and publication order. It allowlists only `MEMORY_LOCAL_DIR`, never loads a Profile environment file or calls Online Memory, and anchors its legacy relative output paths to the invocation directory.
- The Private Text Publisher owns complete staging before commit, strict UTF-8, `0600`, no-follow regular-target checks, atomic replace/create-only publication, cleanup, and explicit partial-publication results. It does not claim a cross-directory transaction; callers own artifact ordering and recovery policy.
- The shared Model Preset Module owns model alias normalization, known-model status, numeric limits, and Pack Mode budgets. Task Pack, Estimate, and Doctor are Adapters at this Seam and must not duplicate model tables.
- The shared Target Project Context Module owns explicit-project canonicalization, Git-root discovery, Profile resolution/loading status, pre-resolved Context consistency, Profile-aware naming, worktree slug policy, and the derived Task Pack path. Command entrypoints retain their own user-facing failure and output policy.
- Target Project Git-root discovery ignores inherited repo-local `GIT_*` variables so a caller's repository context cannot redirect another `--project`.
- Profile Init uses the canonical Target Project root, treats generated Shell values as literal data, preserves relative explicit output paths, and requires explicit `--force` before atomically replacing a private `0600` Profile. Its default project-scoped output refuses a symlinked `.agent-rails` directory and stays anchored to no-follow directory handles.
- The Python Release Builder owns isolated Git tracked/worktree selection, the deterministic single-root kit archive, fixed asset names, SHA-256 manifest, staged validation, and rollback of partial asset publication. CI invokes its absolute Python helper directly. The tag workflow must reject a tag that does not equal `v<VERSION>` before publishing assets.
- The standalone Python Release Installer owns archive download, checksum and path validation, no-link extraction, versioned directory publication, metadata, and atomic `current` / CLI symlink replacement. Its adjacent Shell asset is only a cold-start bootstrap; the installer must reject user-authored non-symlink paths instead of overwriting them and must restore managed state after a failed commit.
- The 11-line top-level bootstrap resolves its physical kit home from its own symlink-aware executable path so stale shell-level state cannot redirect a Release install back to an old source checkout. The Python Public CLI Dispatcher owns the command tree and exports the authoritative home/version to exact child argv without invoking a shell parser.
- The Update Application Service preserves both installation models: Git checkouts use clean fast-forward pull, while Release Installs delegate to the verified installer and re-exec the selected version. `upgrade self` never resolves Target Project Context; project Update resolves Profile path without execution, may run source tests only for Git checkouts, and composes exact Doctor/Adapter child argv.
- Setup, Run, and Verify are Python User Journey Facades entered directly by the Public CLI. Setup passes one pre-resolved Context and exported Profile environment to existing Adapter and Doctor services; Run prepares Pack and estimate results in process; Verify passes one Context to Agent Check and optional Publish Check while taking a fresh Git snapshot after verification succeeds. A non-zero Check result is rendered through the Repair Pack Module before Verify returns the original exit status and skips Publish. Facades must not duplicate Adapter Workspace, Pack Policy, Git Scope, Sensitive Output Guard, Verification Plan, or Repair Pack rules.
- The Python SessionStart Application Service owns host/worktree routing, no-follow Adapter marker discovery, Profile metadata recovery, stable guardrail rendering, and Claude-text/Codex-JSON envelopes. Its 12-line Host Bootstrap Shell only enters the isolated Python helper. SessionStart carries only routing and safety guardrails; task-specific evidence and the full execution contract belong in the on-demand Task Pack.
- The Python Init Application Service owns CLI/environment precedence and literal-safe zsh, bash, and fish guide rendering without writing startup files. The Python Skills Installer owns manifest selection, source-tree validation, complete preflight, and atomic skill-directory publication. Estimate consumes the typed Profile Interface directly; the former shared Paths Shell has no runtime role and is deleted.
- The OpenCode request plugin may use the current session and model input limits to assemble Agent Rails context, but it must not trim or rewrite OpenCode-owned history. Its experimental hook signature requires contract coverage.
- The Context Budget Assembler owns exact final token enforcement, dynamic complete-heading floors, weighted category allocation, unused-share redistribution, complete-line truncation, and balanced fenced blocks. A budget below the actual document's minimum structure must fail rather than emit an empty Pack, a partial heading, or an unclosed fence. Tokenizer implementations are replaceable and optional model-specific dependencies must not become base-install requirements.
- The symlink-aware CLI and SessionStart/Release host bootstraps invoke absolute Python helpers with `-E` or `-I`; the marketplace hook only locates the main host hook. Public commands never route through Shell, and `PYTHONPATH` plus `python -m` from a Target Project is not an allowed runtime seam.
- The Online Memory Interface exposes only read-only search to Task Pack. Agent Rails must not own a provider request schema, credential lookup, write/repair/export workflow, or response JSON parser: the external Online Memory Adapter owns those details and returns only UTF-8 Markdown. The host enforces one deadline, a streaming output cap, whole-process-group cleanup, sensitive-output redaction, and an indented untrusted-data envelope. Adapter failure is non-fatal and falls back to the local memory slice.
- The External Evaluation Harness remains outside the product CLI and Target Project Profile lifecycle. Patch and deterministic acceptance stay primary; OTel/ATIF trajectories are diagnostic evidence.
- A Host Bootstrap Shell remains only where the operating system or host protocol requires a symlink-aware, cold-start, or hook entrypoint. It contains no product policy and remains covered by black-box public and safety contracts. The completed migration ledger lives in `docs/python-refactor-handoff.zh-CN.md`.
- The Terminal Output Module owns common literal/stream safety, Unicode line-boundary normalization, and line/chunk rendering. The Adapter Output Module composes it into one shared stdout/stderr event Interface for Claude, Codex, and OpenCode; each Adapter retains its action, request, result fields, and output policy. The standalone Release Installer duplicates only the cold-start-safe implementation required to run without the package.
- The Child Process Module owns selector-based UTF-8 decoding, pipe cleanup, and the shared Check/Update process-group termination policy. Codex and Online Memory retain distinct bounded-output and deadline policies instead of widening this Interface.
- The test runner owns full and Related Test Selection plus global test setup; each Test Suite owns one coherent workflow area. `--related` maps explicit or current changed paths to ordered suites, while no arguments preserve the full historical gate.
- Shared modules must preserve the public CLI, existing adapter paths, and on-disk compatibility unless a migration is explicitly designed.
